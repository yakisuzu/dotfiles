# System Interface Investigation

- システム境界をまたぐ interface (システム間の契約: 公開 API、メッセージスキーマ、外部公開関数のシグネチャ等) の変更を提案する前に、各システム内部の処理フローを先に調査する
- interface で観測される症状 (欠落・誤値・エラー等) は、内部のフローで情報が落ちている / 誤って変換された結果であるケースが多い
- interface 変更は、調査の結果として真に必要と判明したときの最後の手段

**Why:** システム間 interface は複数の関係者・依存先に影響する共有契約であり、変更コストが高い (相手側の追従、移行、互換維持の調整)。提案段階で安易に interface 変更を出すと、本来直すべき内部の不具合や設計の歪みが温存される

**How to apply:**
- 問題が観測された境界面から、関連システムの内部に向かってフローを辿る
- 各処理段階で「正しい値が来ているか」「どこで失われたか」「どこで意図と違う変換が起きたか」を確認する
- interface 契約の正しさ (契約自体が壊れている) と、内部実装が契約を満たしているか (契約は正しいが実装側のバグ) を区別する
- 内部修正で解決できないと追跡で判明したときのみ、interface 変更を提案する
